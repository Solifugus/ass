## How to resume (read this first)

If you are a fresh Claude Code instance with no memory of prior work:

1. Read this file top to bottom. The **Progress log** (bottom) is the source of truth for what's done.
2. Find the first step whose checkbox is `[ ]` (unchecked) — that is the next step. Steps marked `[x]` are done; `[~]` means in progress (check the Progress log for partial notes).
3. Verify reality before trusting checkboxes: run `go build ./...` and `go test ./...`, and skim the relevant files. If a step is marked done but its acceptance criteria fail, fix that before moving on.
4. Do the step. Keep changes scoped to that step only.
5. When done: change its `[ ]` to `[x]`, append a dated entry to the **Progress log** (what you did, key files, any deviations or decisions), and stop or continue per the user's instruction.

**Rules for keeping steps context-safe:**
- Each step is sized to fit comfortably in one context window. If a step feels too big, split it into sub-steps in place and note that in the Progress log.
- Never start a step without first updating its status to `[~]` if you might not finish it in one session.
- Prefer leaving the tree green (`go build` + `go test` pass) at the end of every step.

**Status legend:** `[ ]` not started · `[~]` in progress · `[x]` done · `[!]` blocked (explain in Progress log)

## Roadmap (post-Phase 13 — the deferral roadmap, foundational-first)

Phases 0–13 are complete, as are the post-roadmap differentiators landed since:
the **resident session model**, the **Jupyter kernel + rich display**, the **PROC
PROOF validation tier**, the **four industry cookbooks**, and the **testing plan**
(see the Progress log and [`testing-plan.md`](testing-plan.md)). What remains is
the set of known deferrals — every one of them is captured below.

The sequencing principle is **foundational-first** (decided 2026-06-25): the goal
is to complete *every* item, so order to make finishing the whole set least
painful — dependency order, so each layer is built once on solid ground, not on
workarounds that later need rework. Language/format *primitives* and I/O
*completeness* land before the higher-level reshaping, reporting, and statistics
that build on them. **The single most leveraged foundation is the testing &
verification base (Phase 13.5) — established first**, because every phase after it
is cheaper and safer to verify; the testing tracks then keep running alongside
Phases 14–18. Note: ordering changes *when* things land, not the total timeline —
the timeline is set by *scope*, almost all of it in Phase 17 (stats).
Architecture/CGo/Jupyter rationale is in [`design.md`](design.md) §14–16.

**Statistics validation (clean-room — no licensed SAS).** Expected values for the
Phase 17 statistical tier are sourced in this precedence order: (1) **published /
textbook** worked examples and public datasets with known answers; (2) **cross-check
against other engines** (R, Python statsmodels) for the same model; (3)
**common-sense / first-principles hand-derivation**. Never proprietary SAS docs,
source, or internals.

### Phase 13.5 — Testing & verification foundation (do this first)
Establish before Phase 14; the testing tracks then keep running alongside every
later phase. This is the most leveraged investment — it de-risks and cheapens
everything after it, and the differential harness in particular is what makes the
Phase-17 stats tier tractable.
- [ ] **Corpus value-verified backfill** — raise `expected.datasets` coverage from 35/63 toward full reference-surface coverage; add a coverage report (extend `ass test --json`) that flags any reference.md surface item lacking a value-verified corpus item, so the backlog is visible and shrinks measurably
- [ ] **Cross-engine differential harness** — automate checking ASS output against R / Python statsmodels (and published values) for the same program, per the validation precedence; cite source/engine+version per item
- [ ] **Edge-case / robustness + fuzz (start)** — missing-value / type-coercion / format-boundary suites and a lexer/parser fuzz target; the rest of this track continues alongside later phases

### Phase 14 — Language & format primitives (foundation for reporting)
- [x] **`PUT()` / `INPUT()` functions** — done 2026-06-25: `put(value, format.)` → value through a user/built-in format (char); `input(string, informat.)` → string through a user/built-in informat (value). Parser captures the format-spec arg verbatim; catalogs threaded via the PDV. Corpus `put_input_functions_001` + unit tests.
- [x] **`intck` / `intnx` advanced interval forms** — done 2026-06-25: unified grid model handles multipliers (`month2`), shifts (`week.2`), `dt`-prefixed datetime intervals (`dtday`/`dtmonth`), the `semiyear` interval, and sub-day intervals (`hour`/`minute`/`second`). Corpus `interval_advanced_001` + `TestIntervalAdvanced`.
- [x] **`proc format` PICTURE templates** — done 2026-06-25: digit-selector templates (zero-suppressing `0` vs forcing nonzero) with literal message characters, default/`mult=` scaling, and `prefix=`/`fill=` options. Rendered in `table.FormatRange.renderPicture`, applied in PROC PRINT and via `put()`. Corpus `proc_format_picture_001` + `table.TestPictureFormat`/`parser.TestParsePictureStatement`.
- [x] **User formats applied to PROC SQL output columns** — done 2026-06-25: a query result column carried through from a source column (same name) inherits that column's format, so a bare `select` listing renders user/built-in formats like PROC PRINT. `proc.inheritSourceFormats` + `TestInheritSourceFormats`.
- [x] **Multi-line PROC PRINT label-header wrapping** — done 2026-06-25: a label wider than its data column wraps across several header lines, split at blanks, with all columns' header blocks bottom-aligned. `proc.wrapHeader` + the multi-row header renderer in `renderListing`. Single-word headers (variable names, short labels) are unchanged. Completes Phase 14.

### Phase 15 — I/O & data-access completeness
- [x] **`infile` / `file` options tail** — done 2026-06-25: INFILE `lrecl=` (record-length cap), `pad` (blank-pad short records to lrecl for column input), `end=var` (last-record flag, temporary/excluded from output); FILE `mod` (append). Parser + runtime (`readInfile` lrecl/pad, `setInfileEnd`, `flatfile.AppendLines`). Corpus `infile_end_001` + parser/runtime tests.
- [ ] **DB pushdown** — push joins/aggregation to the database; `obs=` / `firstobs=` pushdown (value-safe subset already pushed for `keep=` and equality/`>`/`>=` `where=`)
- [ ] **big-endian `.sas7bdat` read** (little-endian, compressed & not, already read)
- [ ] **`.sas7bdat` write**
- [ ] **on-disk format catalogs** (`cntlin`/`cntlout`, persisted user formats)

### Phase 16 — Reshaping & reporting (build on the Phase-14 primitives)
- [ ] **PROC TRANSPOSE** — wide↔long reshaping (`by`/`var`/`id`/`prefix`); the reflexive SAS idiom every analyst reaches for
- [ ] **PROC REPORT** — computed columns, `n (%)` display tables, grouping/ordering/summary lines
- [ ] **PROC TABULATE** — multi-dimensional summary tables
- [ ] **PROC FREQ** — default (non-`list`) stratified n-way layout + association stats beyond Pearson chi-square (likelihood-ratio, Fisher exact, measures of association)

### Phase 17 — Statistical tier (validated per the precedence above)
Committed scope: GLM-fidelity + logistic + survival. Mixed models is **demand-gated**
(below) so it doesn't silently pad the schedule. This phase is the multi-month bulk
of the whole roadmap.
- [ ] **PROC GLM SAS-fidelity** — sweep/generalized-inverse parameterization, Type I/III SS, F-tests, LSMEANS/CONTRAST/ESTIMATE (CLASS via reference-cell coding already fits correctly)
- [ ] **Logistic regression** (PROC LOGISTIC / GENMOD-style)
- [ ] **Survival** — PROC LIFETEST (Kaplan–Meier) and PROC PHREG (Cox PH)
- [ ] **Mixed models** (PROC MIXED) — **demand-gated, not auto-scheduled**: the worst cost/benefit in the tier; build only when a concrete user requires it
- [ ] **PROC PROOF statistical tier** + `abort` (fail-fast) — see [`proofing.md`](proofing.md) §11

### Phase 18 — Interactive / Jupyter polish
- [ ] **Centered titles/footnotes** (SAS centers; ASS currently left-aligns)
- [ ] **Tab-completion (`complete_request`) and introspection (`inspect_request`)**
- [ ] **Cooperative interrupt** — abort a running step on `interrupt_request`
- [ ] **`stdin` / `input_request` round-trip**

### Independent / continuous (not blocking the phases)
- [ ] **Testing tracks (continuous, after Phase 13.5)** — once the testing foundation lands (Phase 13.5), keep backfilling the corpus toward full reference-surface coverage and adding industry end-to-end items, edge/robustness + fuzz, and differential checks as each later phase adds surface. See [`testing-plan.md`](testing-plan.md)
- [ ] **Slot-indexed PDV + non-map rows** (perf foundation; `perf.md`) — resolve vars to slice slots, drop the per-row `map[string]Value`; foundation for any future VM/vectorized executor. Gate on a ≫100k-row workload
- [ ] **Bytecode VM** (`vm/`) — **deferred (evidence-based)**: the 2026-06-24 profile shows dispatch isn't the cost; revisit only after the slot-indexed PDV, if profiling then warrants it
- [ ] (far) native ASS SQL executor on the shared core — wants the VM done first
- [ ] **SAS→ASS migration linter** — reuses the existing SAS parser
- [ ] **Cross-platform release artifacts** (Linux/Windows/macOS ~free via Go; s390x validated; AIX is the hard target)
- [ ] **ODBC LIBNAME engine** (opt-in CGo, like DB2)
- [ ] **Vision layer** (`future-directions.md`): grounded AI assistant; Python/R bridges (Arrow); proofing tiers 2–3 (`constrain`/enhanced `error`, attached constraints, statistical checks)

**Already done since Phase 13** (full detail in the Progress log): date/time functions; modernc.org/sqlite swap (CGo-free default build); DATA-step benchmark + perf win; PROC PROOF validation tier; resident session model; Jupyter kernel + rich display; TITLE/FOOTNOTE; industry cookbooks; testing plan; and the rename-statement / two-way-FREQ-options / MEANS sum+maxdec fixes.

### 2026-06-20 — `PROC IMPORT`/`PROC EXPORT` (delimited) — flat-file objective Phase 3

A dataset round-trips through CSV with values intact (embedded comma quoted on write, unquoted on read; numeric columns recovered). `dbms=csv`/`tab`/`dlm` and `delimiter=`/`dlm=` select the separator.

### 2026-06-19 — Column/pointer input and output — flat-file objective Phase 4
- What changed: INPUT and PUT now support fixed-column and pointer-based field placement, not just delimited list I/O. ### 2026-06-20 — Native `.sas7bdat` reader + base/directory LIBNAME engine
- What changed: ASS can now read native SAS dataset files. `libname lib "/dir"; set lib.member;` (or `proc print data=lib.member;`) reads `dir/member.sas7bdat` as a dataset — the previously-flagged highest single adoption lever, since it lets ASS ingest existing SAS datasets directly rather than only CSV/DB sources. New self-contained package `sas7bdat` with a `Read(path) (*table.Dataset, error)` plus a `DirBackend` implementing `table.Backend`.
- Clean-room constraint: the binary format is implemented strictly from the **public** reverse-engineering literature (the layout documented by the ReadStat and sas7bdat open-source projects and Matthew Shotwell's published format notes) — never from proprietary SAS documentation, source, branding, or internals.
- Method: built with an empirical feedback loop against real fixtures (public-domain `.sas7bdat` samples fetched from the pandas test corpus, several with paired CSVs for independent value verification). Each format layer (file header → page/subheader geometry → column metadata → row data) was decoded, dumped, and checked against the CSV/known values before moving on. Key gotchas pinned down this way: numerics may be **truncated** to <8 bytes (high-order bytes stored; reconstruct by high-byte placement — airline YEAR is a 4-byte double); mix-page row data starts at `bitOffset+8+subCount*ptrLen` rounded up to the next 8-byte boundary (the format's pointer-length invariant makes `base + base%8` always 8-aligned); the header sizing block (headerLength/pageLength/pageCount) shifts by `align1` (byte 35 == '3'), not by the u64 flag (byte 32 == '3'); column-text offsets are relative to the text subheader *after* its signature; format/label refs sit at byte offsets 34/36/38 (format) and 40/42/44 (label) within the format subheader (32-bit), with width@12/ndec@14 → `dollar12.2`, `$char10.`, `monname3.`.
- Scope: 32-bit and 64-bit **little-endian, uncompressed** files; numeric (incl. truncated), character, and SAS date/datetime values (numeric days/seconds from 1960-01-01); column names/lengths/formats/labels (labels/formats recovered for 32-bit; on 64-bit the values read correctly but format/label recovery is skipped — the leading integer block is wider and unverified without a 64-bit fixture). Compressed files (`SASYZCRL` RLE / `SASYZCR2` RDC) are **detected and rejected with a clear error** rather than read partially; big-endian and writing are out of scope.
- Integration: `runLibname` now routes an empty/`base`/`v9`/`v8`/`sas7bdat` engine to `sas7bdat.OpenDir` (a base/directory library) instead of `dbio.Open`; everything else still goes to the database engines. Because `set` and PROC `data=` already resolve external librefs via `table.Library.Resolve`, no DATA-step or PROC change was needed — `set lib.member;`, `where`, `keep`, and `proc print data=lib.member;` all work through the backend.
- Files: `sas7bdat/sas7bdat.go` (header/page/subheader/row decode + `Read`), `sas7bdat/library.go` (`DirBackend`, `OpenDir`), `sas7bdat/reader_test.go` (CSV-verified airline + productsales, test1 structure, datetime SAS-date values, compressed-file error), `sas7bdat/testdata/*` (public-domain fixtures + CSV pairs), `runtime/program.go` (base-engine routing, `isBaseEngine`). ### 2026-06-19 — `.sas7bdat` row decompression: RLE (SASYZCRL) + RDC (SASYZCR2)
- What changed: the `.sas7bdat` reader now reads **row-compressed** files, not just uncompressed ones — the previously-flagged highest-value sas7bdat follow-up, since large real-world SAS datasets are almost always stored compressed. Both SAS-native schemes are supported: `SASYZCRL` (Run Length Encoding) and `SASYZCR2` (RDC, Ross Data Compression). The earlier clean error on compression is gone.
- Clean-room constraint: the decompressors are ports of the **public** reverse-engineering literature only — ReadStat's documented RLE command table and the sas7bdat R-package vignette for RLE, and Ed Ross's published RDC algorithm (Dr. Dobb's Journal, 1992) for RDC — never from proprietary SAS material. The two algorithms were cross-checked against pandas' open-source reference implementation and verified byte-exact against fixtures.
- How compressed files differ: in a compressed file each observation is stored as its own **row subheader** (compression flag `0x04`, type `1`) rather than contiguously after the page's pointer array. A row already at full `rowLength` width was stored verbatim (not worth compressing); a shorter one expands to exactly `rowLength` bytes. `parse()` now records the compression method from the column-text block and, when set, walks every page's subheader pointers extracting/decompressing data subheaders (`readCompressedRows`); the uncompressed contiguous path is unchanged. `decodeRow`/`f64` were refactored to decode from a standalone row byte-slice so both paths share them.

The manual decode of `40 20 30 e0 …` matched the algorithm exactly before coding.
- Files: `sas7bdat/decompress.go` (`rleDecompress`, `rdcDecompress`, `decompressRow`), `sas7bdat/sas7bdat.go` (`reader.compression`, `readCompressedRows`, row-slice `decodeRow`/`f64`, text block records compression instead of erroring), `sas7bdat/reader_test.go` (`TestReadUncompressed`/`TestReadRLE`/`TestReadRDC` full CSV checks + `TestReadRLEControlByte`), `sas7bdat/testdata/*` (test2_rle/test3_rdc + test123.csv, 0x40controlbyte.csv). ### 2026-06-20 — Implicit query pushdown (value-safe `keep=`/`where=` to the database)
- What changed: reading an external (database) source with dataset options now pushes a **value-safe subset** of them into the SQL `SELECT` so the database returns less data, instead of always materializing the whole table and filtering locally. `keep=` becomes a column projection; a `where=` of numeric comparisons using `=`/`>`/`>=` (optionally `literal OP column`, AND/OR-composed) becomes a SQL `WHERE`. Everything else — `<`/`<=`/`ne` (which **keep** missing in SAS, unlike SQL), string/function predicates, `drop=`, NOT — is left to the existing local pass.
- Why it's value-safe: the runtime still applies the full dataset options locally (`applyDatasetOptions` unchanged), so pushdown can only reduce transfer, never change results — and the pushed predicate is deliberately limited to operators that select **exactly** the SAS row set. SAS orders a missing value below every number, so `=`/`>`/`>=` exclude missing precisely as SQL's NULL handling excludes NULL; the unsafe operators (which keep missing in SAS) are never emitted. The backend additionally validates each pushed column is numeric (a zero-row `WHERE 1=0` metadata probe) and maps the projection to the table's real column names (case-insensitively), bailing to `SELECT *` on anything unproven.
- Design: dialect-neutral `table.Selection`/`table.Filter` (a tiny AND/OR/Cmp tree) + optional `table.FilterBackend` (`LoadFiltered`) implemented by `dbio.Backend`; `Library.ResolveFiltered` routes a Selection to a FilterBackend (and is a no-op `Resolve` for WORK/non-filter backends). `runtime/pushdown.go` translates `*ast.DatasetOptions` → `table.Selection` (the safe-operator translator, literal-on-left normalization, WHERE-column collection so a projection keeps the WHERE's columns). The three external read sites (DATA-step SET, MERGE, PROC `data=`) switched from `Resolve` to `ResolveFiltered`; `dbio` renders the Filter with per-engine quoting and drops it whole if any referenced column is non-numeric.
- Verification: `runtime.TestTranslateFilter{SafeOps,AndOr}` + `TestPushdownSelectionKeepIncludesWhereCols` pin the safe/unsafe operator boundary and projection logic; `dbio.TestLoadFiltered{Projection,Where,NonNumericNotPushed,MissingColumnFallsBack}` (SQLite) verify the backend SELECT; `proc.TestPushdown{SafeOpEquivalent,UnsafeOpKeepsMissing}` prove end-to-end value-identity over a source containing a missing numeric — `>=` pushed (missing excluded), `<` not pushed (missing kept). New corpus item `db_query_pushdown_001` value-verifies both sides.

### 2026-06-20 — Trailing `@`/`@@` line-hold on INPUT (DATA-step gap)
- What changed: INPUT now honors the trailing line-hold modifiers. `@@` (double trailing-at) holds the physical line **across DATA-step iterations**, so multiple observations are read from one line — and continues across line breaks until the data is exhausted (the classic `input x @@; datalines; 1 2 3 4 5;` → 5 obs). `@` (single trailing-at) holds the line **within the iteration**, so a control value can be read, tested, and the rest of the line read conditionally with a later INPUT (`input kind $ @; if kind='N' then input value; else input label $;`).
- Design: a per-record column cursor + two hold flags on the data step. INPUT execution moved to `execInput`, which reads via a 1-based cursor while a hold is active (`applyListInputFrom` for list input, `applyColumnInputFrom` for column/pointer input — the latter a refactor of the existing `applyColumnInput`) so consecutive reads share the line; the record advances only when the held line is exhausted (`@@`), at the iteration boundary (`@`, released in the read-driven loop), or immediately (no hold, the unchanged whole-line `applyInput` path). The read loop's safety progress-bump is gated so a `@@` hold (cursor progress, not record progress) is not skipped. Parser detects a trailing `@`/`@@` (spaced or attached) and sets `InputStatement.TrailingAt` (1/2); no `@n` pointer ends in `@`, so the two never collide.

### 2026-06-20 — `#n` multi-line line pointers on INPUT and PUT (DATA-step gap)
- What changed: INPUT and PUT now honor `#n` line pointers, so one observation spans several physical lines. On PUT, `#n` writes one observation as several physical lines (`put #1 name age #2 city zip;` emits two lines per observation), so an INPUT/PUT pair round-trips a multi-line record.

### 2026-06-20 — PROC SQL: external libref (and WORK-qualified) tables as query sources
- What changed: ordinary `proc sql` (the in-process SQLite path, not pass-through) can now read a **libref-qualified table as a source** — `select … from db.orders`, a join of an external table with a WORK table, and `create table … as select … from db.x` — and resolves **WORK-qualified sources** (`work.x`). ### 2026-06-20 — Backlog sweep #1: trailing `@`/`@@` output hold on PUT (DATA-step gap)
- What changed: PUT now honors trailing line-hold modifiers, mirroring INPUT. A trailing `@` holds the **output** line within the iteration so successive PUTs build one physical line (`put name @; put age;` → `"Amy 30"`), released automatically at the iteration boundary; a trailing `@@` holds it **across iterations** so several observations write to one line (`put x @@;` over 1/2/3 → `"1 2 3"`), released by a PUT without a trailing hold or at end of step. Previously a trailing `@`/`@@` on PUT was parsed as a stray variable.
- Design: `PutStatement.TrailingAt` (0/1/2); `parsePut` detects/strips a final `@`/`@@` (spaced or attached) from the raw item slice before `parsePutItems` — no `@n` pointer or format ever ends a PUT, and a quoted literal ends in its quote, so a trailing `@` is unambiguously a hold. Runtime PUT execution moved to `execPut`, which keeps a held partial line (`putBuf`) plus `putHeld`/`putHoldX` flags: with a hold active the new segment continues the held line — list segments join by the FILE separator (`listSep`), column/pointer segments `overlayLine` onto the held line at their absolute columns. Single `@` is flushed at the end of `runIteration`; `@@` is flushed at end of step (before the file is written). `#n` multi-line PUT flushes any hold first and emits its own lines (combining the two stays a documented gap, tracked separately).
- Verification: `parser.TestParsePutTrailingAt` (spaced/attached `@`/`@@`, plain list, leading `@n` not misread); `runtime.TestPut{SingleTrailingAt,DoubleTrailingAt,ColumnTrailingAt,NoTrailingAtUnchanged}` cover list single-`@`, cross-iteration `@@`, column-mode overlay, and a default-path regression guard. ### 2026-06-20 — Backlog sweep #6: combine #n line pointer with a trailing @/@@ hold
- What changed: a `#n` multi-line INPUT statement may now carry a trailing `@`/`@@` hold (previously the hold was ignored when `#n` was present). `input a #2 b @@;` holds the multi-line record *group* across iterations with a per-line column cursor, so several observations are read from one group: with two tokens per line, `1 10` / `2 20` yields (1,2) then (10,20); with one token per line the group is exhausted after one observation and the pointer advances to the next group.

### 2026-06-24 — DATA-step performance baseline + the bytecode-VM decision (Track C)
- What changed: established the first DATA-step performance baseline and applied the cheapest high-impact interpreter win, then **recorded the evidence-based decision on the bytecode VM** (the explicit purpose of this roadmap item). New `runtime/bench_test.go` with two 50k-row benchmarks: `SetTransform` (representative `set`+arithmetic+`if/then`+`keep=` ETL path) and `Generate` (compute-bound `do`-loop + `output`). Full write-up in new `docs/perf.md`.
- The win: `writeRow` rebuilt the output column set every row — `table.Dataset.AddColumn` does a linear `strings.ToLower`-ing scan, so output was O(rows × vars²) with a per-row `pdv.Names()` allocation. Now it computes the output-column plan **once** (cached on the dataStep, rebuilt only when the PDV variable count grows) and the per-row path copies values by precomputed lowercased key via a new `PDV.GetLower` (no per-row `ToLower`, no `Names()` alloc). Result: SetTransform ~757→~555 ms (~1.37×), Generate ~473→~315 ms (~1.50×), allocs 150k→100k per op. (Also tried in-place `ResetVars` to avoid map-delete churn — no measurable gain, reverted.)
- Decision (perf.md): the per-row loop *is* the bottleneck (not parse/IO/PROC), satisfying the roadmap's precondition for the VM — **but the profile shows bytecode dispatch is not where the time goes.** After the writeRow fix, cost is GC from the per-row `map[string]Value` row allocation + PDV string-map hashing. So the bytecode VM is **deferred**; the actual next perf lever is a **slot-indexed PDV (registers) + non-map row representation**, which stands alone and would be the foundation for any later VM/vectorized executor. ~90–160k rows/s is adequate for ASS's ETL/reporting targets; breadth + differentiators come first. The tree-walk interpreter stays the corpus-validated correctness oracle.

### 2026-06-24 — Resident session model + `ass repl` (keystone for interactivity)
- What changed: turned the batch runner ("parse file → run → exit") into a long-lived interpreter. New `session` package: `session.New()` holds a persistent `table.Library` + `macro.Processor`, and `(*Session).Submit(src, logger)` macro-expands → parses → runs one program fragment against that shared state. Datasets, librefs, macro variables, and macro definitions all carry across submissions, exactly as if the fragments had been one file. This is the `design.md` §16 keystone the Jupyter kernel builds on.
- Design: `session` imports `runtime`/`parser`/`macro`/`table`/`log`; nothing imports it back, so no cycle (and it keeps `parser`/`macro` out of `runtime`'s production deps). Parse failures return a typed `*session.ParseError` (fragment runs nothing, state untouched); runtime errors pass through; a step that only *logs* errors (e.g. failing PROC PROOF) returns nil and is gated via `logger.ErrorCount`, same as batch. `cmd/ass` `runProgram` is now literally `session.New().Submit(file, logger)` — the batch path is the one-submission degenerate case.
- New consumer: `ass repl` — an interactive session reading SAS from stdin into one resident `Session`. A fragment submits on a `run;`/`quit;` line or a blank line (flushing buffered globals like `%let`/`libname`); `endsas;` or Ctrl-D exits. Proven end-to-end: a `%let` in one entry feeds a `&ref` in a later DATA step, and a dataset built in one entry is `set` by another.

### 2026-06-25 — Phase 14: PUT() / INPUT() functions
- `put(value, format.)` renders a value through a format (user VALUE format from the catalog, else built-in) returning a character string; `input(string, informat.)` reads a string through an informat (user INVALUE, else built-in) returning the value. The canonical enabler for format-based banding (`band = put(score, scoreband.)`) — replaces the workarounds the cookbooks used.
- The second argument is a format/informat spec (`dollar8.2`, `comma8.`, `agegrp.`) that doesn't tokenize as an expression, so the parser special-cases `put`/`input` (`parser.parsePutInput`) and captures the spec verbatim from source as a string-literal arg. The format/informat catalogs are threaded to expression eval via two new PDV fields (`formats`/`informats`), set from the library at DATA-step start — no rethreading of `Eval`.
- `runtime.applyPutFormat` / `applyInputInformat` reuse the existing `formats.Apply` / `formats.ParseInput` and the user catalogs (mirrors `proc.applyFmt`).
- Verification: `runtime.TestEvalPutInput` (user format, built-in format, comma + date9 informats) + corpus `put_input_functions_001` (banding + comma informat, value-verified). gofmt/build/vet/test clean both CGO modes; corpus **67/67**, value-verified **47/47**; coverage unchanged at 2 gaps (proc-reg/glm).

### 2026-06-25 — Phase 14: intck / intnx advanced interval forms
- Replaced the per-interval switch in `intck`/`intnx` with a **unified grid model** (`runtime.intervalSpec`): every interval partitions a one-dimensional grid (months, days, or fixed-size seconds) into runs of `period` grid units offset by `offset` units. `intck` = difference of run ordinals; `intnx` = jump `n` runs then align (`b`/`m`/`e`/`s`). ### 2026-06-25 — Phase 14: PROC FORMAT PICTURE templates
- `picture name range='template' (options)` defines an output picture format. The template is a string of **digit selectors** (0-9) and **message characters**. Digit selectors are positions the value's digits fill right-to-left; a `0` selector zero-suppresses leading positions (printed as the fill char, blank by default) while a nonzero selector forces printing from its position rightward. Message characters (comma, dot, dash, %, …) print only once significant digits have begun. The value is scaled by a multiplier — default `10^(selectors after the decimal point)`, or an explicit `mult=` — and rounded before mapping. Options: `prefix=` (inserted before the first printed digit), `mult=`, `fill=`. Sign is dropped (SAS picture behavior).
- AST: `ValueStatement.Picture` + per-range `Prefix`/`Mult`/`Fill`. Parser: new `picture` case reusing `parseValueStmt` with a `picture` flag, plus `parsePictureOptions` for the `(…)` list. table: `ValueFormat.Picture` + `FormatRange.{Prefix,Mult,Fill}` and `renderPicture`, dispatched from `ValueFormat.Format` via a new `render` helper (so PROC PRINT, MEANS/FREQ grouping, and `put()` all pick it up unchanged). proc/format.go builds picture formats and copies options.
- Verification: `table.TestPictureFormat` (currency with comma grouping + `$` prefix + default mult 100; phone all-suppress + dash; percent default mult 10 + trailing `%`; explicit `mult=1 fill='0'` zero-pad), `parser.TestParsePictureStatement`, and corpus `proc_format_picture_001` (value-verified via `strip(put(...))` to avoid picture-width brittleness). gofmt/build/vet/test clean both CGO modes; corpus **69/69**, value-verified **49/49**.
- Docs: reference.md (PICTURE in the PROC FORMAT section, deferral removed), COMPATIBILITY.md, README, CLAUDE.md updated.

### 2026-06-25 — Phase 15 started: infile/file options tail (lrecl/pad/end/mod)
- **INFILE `lrecl=`** caps the record length (longer lines truncated). **`pad`** blank-pads short records out to `lrecl`, so fixed-column INPUT past a short line reads missing rather than erroring (e.g. `input code $ 1-2 num 3-5` over a 2-char line yields `num=.`). **`end=var`** names a flag set to 1 on the last record (0 before), the canonical idiom for emitting a single summary row (`if last then output;`); the flag is temporary — added to the step's drop set so it never reaches the output dataset. **FILE `mod`** appends to the file instead of overwriting.
