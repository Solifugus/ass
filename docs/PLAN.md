# ASS Development Plan

Living plan for building Analyst's Statistical Suite (ASS), a SAS-compatible ETL/analytics engine in Go. See [`design.md`](design.md) for the design rationale and [`../CLAUDE.md`](../CLAUDE.md) for architecture notes.

## How to resume (read this first)

If you are a fresh Claude Code instance with no memory of prior work:

1. Read this file top to bottom. The **Progress log** (bottom) is the source of truth for what's done.
2. Find the first step whose checkbox is `[ ]` (unchecked) — that is the next step. Steps marked `[x]` are done; `[~]` means in progress (check the Progress log for partial notes).
3. Verify reality before trusting checkboxes: run `go build ./...` and `go test ./...`, and skim the relevant files. If a step is marked done but its acceptance criteria fail, fix that before moving on.
4. Do the step. Keep changes scoped to that step only.
5. When done: change its `[ ]` to `[x]`, append a dated entry to the **Progress log** (what you did, key files, any deviations or decisions), and stop or continue per the user's instruction.

**Rules for keeping steps context-safe:**
- Each step is sized to fit comfortably in one context window. If a step feels too big, split it into sub-steps in place and note that in the Progress log.
- Never start a step without first updating its status to `[~]` if you might not finish it in one session.
- Prefer leaving the tree green (`go build` + `go test` pass) at the end of every step.

**Status legend:** `[ ]` not started · `[~]` in progress · `[x]` done · `[!]` blocked (explain in Progress log)

## Phase 2 — Lexer

- [x] **2.1 Token types.** In `lexer/token.go`, define `TokenType` constants and a `Token` struct (type, literal, line, col). Cover: identifiers, numbers, string literals (single+double quote), `;`, `=`, comparison/arith operators, `(` `)` `,` `.` `&` `%`, comment markers (`*...;` and `/* */`), and keyword detection helper. Acceptance: compiles; constants documented.
- [x] **2.2 Core scanner.** In `lexer/lexer.go`, implement a `Lexer` that scans a string into tokens: whitespace skipping, line/col tracking, identifiers, numbers, strings, single-char tokens. Add `lexer/lexer_test.go` with table tests. Acceptance: `go test ./lexer/` passes for basic tokens.
- [x] **2.3 Comments & SAS quirks.** Handle SAS's two comment forms (`* ... ;` statement comments and `/* ... */` block comments) and the `$` character-variable marker. Add tests. Acceptance: comment tests pass; comments produce no spurious tokens.
- [x] **2.4 Step/keyword recognition.** Recognize statement-leading keywords enough to detect `data`, `proc`, `run`, `quit`, `datalines`/`cards`, and handle the special raw-data region after `datalines;` up to a line with `;`. Add tests for a full `data ... datalines ... run;` token stream. Acceptance: datalines region tokenized correctly (raw lines, not re-lexed as code).
- [x] **2.5 Wire lexer into CLI.** Make `ass <file.sas>` print the token stream (debug mode) so the lexer is exercisable end-to-end. Acceptance: running it on a Level-1 corpus item prints sensible tokens.

## Phase 3 — Parser & AST

- [x] **3.1 AST node definitions.** In `ast/`, define interfaces (`Node`, `Statement`, `Expression`) and the program/step containers: `Program` (list of steps), `DataStep`, `ProcStep` (generic with proc name + raw options/statements for now). Acceptance: compiles.
- [x] **3.2 Statement & expression nodes.** Add nodes for: assignment, `set`, `input` (with var list + `$` flags), `datalines` (raw block), `if/then/else`, `do/end`, `output`, `keep/drop`, plus expression nodes (literal, identifier, binary op, unary op, function call). Acceptance: compiles; nodes have `String()` for debugging.
- [x] **3.3 Program/step parser.** In `parser/parser.go`, implement top-level parsing that splits the token stream into steps by `data`/`proc` ... `run;`/`quit;` and dispatches to step parsers. Add `parser/parser_test.go`. Acceptance: parses a multi-step file into a `Program` with the right step count/types.
- [x] **3.4 Expression parser (Pratt).** Implement precedence-climbing/Pratt parsing for SAS expressions: arithmetic (`+ - * / **`), comparison (`= ^= < <= > >=` and word forms `eq ne lt`...), logical (`and or not`), function calls, parentheses. Add tests. Acceptance: expression tests pass including precedence.
- [x] **3.5 DATA step statement parser.** Parse the bodies of DATA steps into the statement nodes from 3.2. Add tests against Level-1 corpus items. Acceptance: Level-1 items parse to expected AST shape.
- [x] **3.6 PROC step option parser.** Parse common PROC option syntax: `proc <name> data=<ds> (options);` and statement lines like `by`, `var`, `where`. Keep proc-specific semantics out (just structure). **Caveat from Phase 2:** the lexer emits `data` as the `DATA` keyword token even in `proc print data=people` (SAS keywords are contextual). The option parser must accept the `DATA` token (and likely `SET`, etc.) where an option name is expected, rather than only `IDENT`. Acceptance: PROC PRINT/SORT items parse.
- [x] **3.7 Wire parser into CLI.** Add `ass parse <file.sas>` (or a flag) that prints the AST. Acceptance: prints AST for corpus items.

## Phase 4 — DATA step runtime (the core)

- [x] **4.1 Dataset model.** In `table/`, implement `Dataset` (library name, name, ordered columns with name/type/label/format/informat, rows as `[]Row`), `Column`, and a `Value` type supporting numeric, character, and **missing** (`.` numeric, `''` char). Add tests for construction and missing-value semantics. Acceptance: `go test ./table/` passes.
- [x] **4.2 In-memory library/catalog.** Implement a `Library` map (e.g. `work`) holding datasets by name, with get/put. This is how steps pass data. Acceptance: tests for put/get and overwrite.
- [x] **4.3 PDV + expression evaluator.** In `runtime/`, implement the Program Data Vector (variable name → current Value) and an expression evaluator over the PDV (numbers, strings, missing propagation, arithmetic, comparison returning 1/0, logical ops, a few core functions like `sum`, `upcase`). Add tests. Acceptance: evaluator tests pass including missing-value propagation rules.
- [x] **4.4 Implicit loop & assignment/output.** Implement the DATA step driver: initialize PDV, run the implicit row loop, execute assignment statements, handle automatic `_N_` (iteration counter) and `_ERROR_`. For a step with no input, run once. Implement explicit/implicit `output`. Acceptance: a `data` step with assignments produces the expected dataset.
- [x] **4.5 `input` + `datalines`.** Implement reading the raw datalines region per the `input` spec (list input: space-delimited, `$` = char). Populate the PDV and output one row per data line. Acceptance: the canonical `data people; input name $ age; datalines; ... run;` produces a 2-row dataset.
- [x] **4.6 `set` (read existing dataset).** Implement `set <ds>`: the implicit loop iterates rows of the input dataset into the PDV. Acceptance: `data b; set a; run;` copies dataset a to b.
- [x] **4.7 `if/then/else` + subsetting if.** Implement conditional execution and the subsetting-`if` (a bare `if cond;` that drops the row when false). Acceptance: `data adults; set people; if age>=18; run;` filters correctly.
- [x] **4.8 `do/end` loops & `keep`/`drop`.** Implement iterative `do`/`do while`/`do until` and column selection via `keep`/`drop` statements (and dataset options if feasible). Acceptance: tests for a do loop and keep/drop pass.
- [x] **4.9 SAS-style log output.** In `log/`, implement a logger that writes SAS-like NOTEs (e.g. "NOTE: The data set WORK.PEOPLE has N observations and M variables."). Wire DATA step to emit these. Acceptance: log lines match the expected format for Level-1 items.

## Phase 5 — PROC PRINT

- [x] **5.1 PROC dispatch.** In `proc/`, define a `Proc` interface (run against a Library + step AST + logger) and a registry mapping proc name → implementation. Wire the runtime to dispatch PROC steps. Acceptance: unknown procs produce a clean "not supported" log note, not a crash.
- [x] **5.2 PROC PRINT core.** Implement `proc print data=<ds>;` rendering a SAS-like listing (Obs column + variables, right/left alignment by type). Support `var` to select/order columns. Add tests comparing rendered text to expected. Acceptance: PROC PRINT renders match hand-written expected listings (unit tests in `proc/`). **DECISION (2026-06-16):** the corpus `expected_output.txt` → `verified` backfill (originally folded in here / step 1.6) is **deferred** and NOT done from the ASS engine's own output. Per `corpus/README.md`, `verified` means *hand-derived from real SAS*, which this environment can't produce; capturing our own listing and calling it verified would mislabel it. Backfill is blocked on (a) access to real SAS output, or (b) a Phase 11 decision on whether ASS's clean-room listing format is itself the comparison target (and an added `regression`/`baseline` output state if so). Items stay `output: unverified` until then. New tracking step 11.x to be added in Phase 11.
- [x] **5.3 PROC PRINT options.** Add `noobs`, `label` (use column labels as headers). Acceptance: tests for noobs/label pass.

### 2026-06-19 — Column/pointer input and output — flat-file objective Phase 4
- What changed: INPUT and PUT now support fixed-column and pointer-based field placement, not just delimited list I/O. ### 2026-06-20 — Native `.sas7bdat` reader + base/directory LIBNAME engine
- What changed: ASS can now read native SAS dataset files. `libname lib "/dir"; set lib.member;` (or `proc print data=lib.member;`) reads `dir/member.sas7bdat` as a dataset — the previously-flagged highest single adoption lever, since it lets ASS ingest existing SAS datasets directly rather than only CSV/DB sources. New self-contained package `sas7bdat` with a `Read(path) (*table.Dataset, error)` plus a `DirBackend` implementing `table.Backend`.
- Clean-room constraint: the binary format is implemented strictly from the **public** reverse-engineering literature (the layout documented by the ReadStat and sas7bdat open-source projects and Matthew Shotwell's published format notes) — never from proprietary SAS documentation, source, branding, or internals.
- Method: built with an empirical feedback loop against real fixtures (public-domain `.sas7bdat` samples fetched from the pandas test corpus, several with paired CSVs for independent value verification). Each format layer (file header → page/subheader geometry → column metadata → row data) was decoded, dumped, and checked against the CSV/known values before moving on. Key gotchas pinned down this way: numerics may be **truncated** to <8 bytes (high-order bytes stored; reconstruct by high-byte placement — airline YEAR is a 4-byte double); mix-page row data starts at `bitOffset+8+subCount*ptrLen` rounded up to the next 8-byte boundary (the format's pointer-length invariant makes `base + base%8` always 8-aligned); the header sizing block (headerLength/pageLength/pageCount) shifts by `align1` (byte 35 == '3'), not by the u64 flag (byte 32 == '3'); column-text offsets are relative to the text subheader *after* its signature; format/label refs sit at byte offsets 34/36/38 (format) and 40/42/44 (label) within the format subheader (32-bit), with width@12/ndec@14 → `dollar12.2`, `$char10.`, `monname3.`.
- Scope: 32-bit and 64-bit **little-endian, uncompressed** files; numeric (incl. truncated), character, and SAS date/datetime values (numeric days/seconds from 1960-01-01); column names/lengths/formats/labels (labels/formats recovered for 32-bit; on 64-bit the values read correctly but format/label recovery is skipped — the leading integer block is wider and unverified without a 64-bit fixture). Compressed files (`SASYZCRL` RLE / `SASYZCR2` RDC) are **detected and rejected with a clear error** rather than read partially; big-endian and writing are out of scope.
- Integration: `runLibname` now routes an empty/`base`/`v9`/`v8`/`sas7bdat` engine to `sas7bdat.OpenDir` (a base/directory library) instead of `dbio.Open`; everything else still goes to the database engines. Because `set` and PROC `data=` already resolve external librefs via `table.Library.Resolve`, no DATA-step or PROC change was needed — `set lib.member;`, `where`, `keep`, and `proc print data=lib.member;` all work through the backend.
- Files: `sas7bdat/sas7bdat.go` (header/page/subheader/row decode + `Read`), `sas7bdat/library.go` (`DirBackend`, `OpenDir`), `sas7bdat/reader_test.go` (CSV-verified airline + productsales, test1 structure, datetime SAS-date values, compressed-file error), `sas7bdat/testdata/*` (public-domain fixtures + CSV pairs), `runtime/program.go` (base-engine routing, `isBaseEngine`). Corpus item `proc_print_sas7bdat_001` (+ airline/productsales fixtures) value-verifies reads through the LIBNAME engine; FEATURES `sas7bdat`/`libname` tags.
- Verification: `go build`/`go vet`/`go test ./...` clean; `ass test corpus/` 38/38 (100%), value-verified 12/12. Unit tests verify full datasets against paired CSVs (numeric incl. truncated, character) and hand-derived SAS-date values.

Other levers unchanged: DB write-back, JSON harness output.

### 2026-06-19 — `.sas7bdat` row decompression: RLE (SASYZCRL) + RDC (SASYZCR2)
- What changed: the `.sas7bdat` reader now reads **row-compressed** files, not just uncompressed ones — the previously-flagged highest-value sas7bdat follow-up, since large real-world SAS datasets are almost always stored compressed. Both SAS-native schemes are supported: `SASYZCRL` (Run Length Encoding) and `SASYZCR2` (RDC, Ross Data Compression). The earlier clean error on compression is gone.
- Clean-room constraint: the decompressors are ports of the **public** reverse-engineering literature only — ReadStat's documented RLE command table and the sas7bdat R-package vignette for RLE, and Ed Ross's published RDC algorithm (Dr. Dobb's Journal, 1992) for RDC — never from proprietary SAS material. The two algorithms were cross-checked against pandas' open-source reference implementation and verified byte-exact against fixtures.
- How compressed files differ: in a compressed file each observation is stored as its own **row subheader** (compression flag `0x04`, type `1`) rather than contiguously after the page's pointer array. A row already at full `rowLength` width was stored verbatim (not worth compressing); a shorter one expands to exactly `rowLength` bytes. `parse()` now records the compression method from the column-text block and, when set, walks every page's subheader pointers extracting/decompressing data subheaders (`readCompressedRows`); the uncompressed contiguous path is unchanged. `decodeRow`/`f64` were refactored to decode from a standalone row byte-slice so both paths share them.

The manual decode of `40 20 30 e0 …` matched the algorithm exactly before coding.
- Files: `sas7bdat/decompress.go` (`rleDecompress`, `rdcDecompress`, `decompressRow`), `sas7bdat/sas7bdat.go` (`reader.compression`, `readCompressedRows`, row-slice `decodeRow`/`f64`, text block records compression instead of erroring), `sas7bdat/reader_test.go` (`TestReadUncompressed`/`TestReadRLE`/`TestReadRDC` full CSV checks + `TestReadRLEControlByte`), `sas7bdat/testdata/*` (test2_rle/test3_rdc + test123.csv, 0x40controlbyte.csv). ### 2026-06-20 — PROC APPEND (WORK + database libref, in-place INSERT)

### 2026-06-20 — PROC SQL explicit pass-through (`connect to` / `from connection to` / `execute` / external `drop`)
- What changed: PROC SQL can now send an external database its **own native SQL** instead of routing everything through the in-process SQLite engine. Supported statements: `connect to <engine> [as <alias>] (connection="…")` (opens a block-scoped connection; `dsn=`/`connect=` accepted as synonyms), `disconnect from <alias>`, `select … from connection to <name> (<native query>)` (runs the query on the DB and brings the result set back as a dataset — bare select prints, `create table <tgt> as …` materializes via the shared `Library.Store`), and `execute (<native sql>) by <name>` (remote DDL/DML, no result). A libref already assigned with `libname` can be named directly in `from connection to`/`execute` with no `connect to` (its connection is reused and left open). This also closes the long-standing wart: `proc sql; drop table <libref>.<member>;` is now routed to the external table (via `Library.DropExternal`) instead of hitting embedded SQLite.
- Design: new optional backend interfaces in `table` — `SQLBackend` (`QuerySQL`/`ExecSQL`) and `DropBackend` (`Drop`) — implemented by `dbio.Backend` for every engine (Postgres/SQL Server/Oracle/SQLite); `Load`'s row-scan was extracted to a shared `scanResult` reused by `QuerySQL`. Pass-through dispatch lives in `proc/sql_passthru.go` (regex-recognized statements over the existing `RawBody`/`splitStatements` flow; no grammar change), with `Library.Backend(libref)` exposing assigned backends and `Library.DropExternal` mirroring the existing `StoreExternal`/`AppendExternal` routing. Native SQL is sent verbatim — ASS does not parse/rewrite it; returned columns map to SAS types exactly as a read does.
- Verification: `dbio.TestPassthrough{QueryExecDrop,BackendInterfaces}` (SQLite, no server) cover `QuerySQL`/`ExecSQL`/`Drop` and the interface assertions; `proc.TestPassthrough{AssignedLibref,ConnectExecuteDrop}` drive the full pipeline (parser→runtime→PROC SQL→dbio) end-to-end over a temp SQLite file — assigned-libref reuse, explicit `connect to`, `execute` DML, result materialization, and external `drop`. New corpus item `proc_sql_passthrough_001` value-verifies a `from connection to` GROUP BY result (East=150, West=70).
- `gofmt`/`go build`/`go vet`/`go test ./...` clean; `ass test corpus/` now **44/44**, value-verified **18/18**.
- Files: `table/library.go` (SQLBackend/DropBackend interfaces, Backend accessor, DropExternal), `dbio/dbio.go` (QuerySQL/ExecSQL/Drop/scanResult), `proc/sql.go` (dispatch), `proc/sql_passthru.go` (new), `dbio/dbio_passthru_test.go` + `proc/sql_passthru_test.go` (new), `corpus/proc_sql_passthrough_001/` (new), docs (`databases.md`, `COMPATIBILITY.md`, `corpus/FEATURES.md`).
